Web Browser

18 May 2012

There are abundant choices for browsing the web these days. It has come to my attention that I should perhaps prepare my site to look and feel pleasant regardless of the visitor's web browser. I usually use Safari and sometimes Firefox, but to be comprehensive I tested Internet Explorer 9.0.8112.16421, Chrome 180.1.1025.40 canary and Opera 11.63 for good measures too.

I knew in advance that the site is standards compliant and should look well and work properly on all major platform's web browsers. The result of my tests revealed that Internet Explorer sadly renders the site worst. Then comes Opera followed closely by Firefox. Chrome (canary) performs comparable to Safari, which is unsurprisingly the best. It should be noted that I develop and test the site using Safari and the fact that it came on top was expected. These empirical results are based completely on how much of the W3 standards are successfully implemented by each of the above mentioned web browsers or rather their respective vendors.

As you may know, Safari and Chrome share the same engine called WebKit, which was developed and open sourced by Apple. WebKit is in active development. Firefox's engine is called Gecko, which contains the remnants of the project started at Netscape in the late 90's. Internet Explorer has a proprietary engine developed internally by Microsoft and Opera's engine is called Presto. The engines I have mentioned so far refer to the underlaying libraries and frameworks that are responsible for laying out the DOM elements of a web page. Each of these browsers comes with its own respective JavaScript engine, but for the time being I'm not going to delve deeper into those details.

The best web browsers right now are WebKit based browsers (like Safari or Chrome), simply because you'll view the sites the exact way their developers have intended you to experience them. There certainly are obvious advantageous and disadvantageous to using any specific web browser, which ultimately boils down to an individual's taste and opinion. I emphasise on using a WebKit browser, because whilst they are the most standards compliant of the bunch, they also provide the most features one could seek. For instance, the elastic scrolling specific to Safari on OS X and iOS is only available on Chrome. You might argue that is not part of W3's standards for a web browser to have such feature. True, but wouldn't you like your web browser to incorporate that feature too? In this very site, I have made extensive use of dynamically loaded lists of posts and designs for saving bandwidth and providing a richer user experience. This works very much like Twitter's or Facebook's endless stream of content, where by reaching the bottom of the page a command is automatically initiated to append more content to that page. Here, you won't be able to see more than the initial posts or designs loaded, if:

  1. your screen resolution is high enough that such initial load of data won't enable the vertical scrollbars on your web browser, and
  2. your web browser lacks the aforementioned elastic scrolling feature.

Other sites rectify this situation by either:

  1. load an initial list of content that is long enough that guarantees visitor's web browser to enable the vertical scrollbars, or
  2. instead of loading more content dynamically via a scroll that reaches the end of the page, rely on a 'more' button or link that visitors are forced to click if they want to see more.

These are unintuitive ways of addressing this issue. I could provide a 'more' button or link for web browsers that lack these features, but I decided to avoid that and write a post to perhaps convince many of you that still use other web browsers why you should try a WebKit based browser. In the not so distant future, I will come back to this topic of web browsers and discuss the various JavaScript engines in more details.
